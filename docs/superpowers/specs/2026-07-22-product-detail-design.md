# 商品详情页设计规格

## 1. 背景

当前 `/catalog` 商品目录只渲染普通商品卡片。图片、标题和卡片本身没有详情链接，右下角按钮只调用 `addToCartAuth()`。项目虽然提供公开 JSON 接口 `GET /api/products/{product}`，但没有 Web 商品详情路由、控制器方法或 Blade 页面，因此用户点击商品不会跳转。

本项目用于毕业设计。本迭代以完成可演示的商品浏览到加购纵向链路为第一优先级，不引入与核心功能无关的复杂能力。

## 2. 目标

提供一个公开、可分享、响应式的商品详情页，使访客可以从商品目录进入详情页，查看完整商品信息、选择有效数量并加入购物车。

## 3. 用户价值

- 用户可以在购买前查看商品的完整信息和当前库存。
- 用户可以选择数量后直接加入现有购物车流程。
- 演示者可以通过固定 URL 展示完整的“目录 → 详情 → 加购”业务链路。

## 4. 功能范围

### 4.1 公开详情路由

- 新增 `GET /products/{product}`。
- 路由无需登录。
- 仅 `status = published` 的商品可访问。
- 商品不存在、为 `draft` 或为 `archived` 时返回标准 404。

### 4.2 商品详情内容

页面必须展示：

- 商品主图；无图片时显示稳定的本地视觉占位区域。
- 商品分类。
- 商品名称。
- HKD 价格。
- 产地。
- 有机标识。
- 碳足迹与 `kg CO2e` 单位。
- 当前库存状态。
- 完整商品描述。
- 返回商品目录入口。

字段缺失时不渲染对应的可选信息块，不显示空标签或 `null`。

### 4.3 数量与加入购物车

- 有库存时显示数量输入控件，初始值为 1。
- 合法数量范围为 1 到当前库存。
- 点击加入购物车时调用现有全局函数：

  ```javascript
  addToCartAuth(productId, productName, productPrice, quantity)
  ```

- 已登录用户继续通过 `/api/cart` 持久化购物车；访客继续使用现有 `localStorage` 回退逻辑。
- 售罄商品显示明确的售罄状态，并禁用数量输入和加入购物车按钮。
- 前端输入约束用于即时反馈，库存最终正确性仍由购物车 API 校验。

### 4.4 商品目录入口

- 商品图片链接到 `/products/{id}`。
- 商品标题链接到 `/products/{id}`。
- 右下角“＋”按钮保持当前直接加购行为。
- 不把整个卡片包成链接，避免链接与按钮形成嵌套交互冲突。

### 4.5 公开 API 一致性

- `GET /api/products/{product}` 与商品列表保持一致，只允许返回 `published` 商品。
- `draft`、`archived` 和不存在商品均返回 404。
- API 响应结构保持 `{"data": product}`，不改变现有客户端契约。

### 4.6 国际化

所有新增用户可见文案必须加入：

- `resources/lang/zh.json`
- `resources/lang/zhhk.json`
- `resources/lang/en.json`

新增文案统一放在 `productDetail` 命名空间下。商品数据库内容保持原值，不在本迭代中进行商品内容翻译。

## 5. 页面设计

### 5.1 桌面端

- 顶部沿用现有 `layouts.app` 导航。
- 主内容为左右双栏：左侧商品图片，右侧商品信息和购买操作。
- 价格、库存与加入购物车操作保持在首屏主要视觉区域。
- 描述放在购买操作下方，不新增标签页。

### 5.2 移动端

- 双栏折叠为单栏，图片在上、信息与购买操作在下。
- 数量输入和加入购物车按钮保持可点击尺寸。
- 页面不新增固定底栏或浮动购买栏，以控制实现范围。

### 5.3 视觉一致性

- 复用现有绿色品牌色、圆角卡片、阴影、Lucide 图标和 Tailwind 工具类。
- 不引入新的 CSS 框架、JavaScript 框架或组件库。

## 6. 架构与数据流

1. 浏览器请求 `GET /products/{product}`。
2. Web `ProductController::show(Product $product)` 校验商品为 `published`，加载分类关系并返回 `shop.product-detail`。
3. Blade 直接渲染商品模型及其 `image_url` 访问器。
4. 用户调整数量并点击加入购物车。
5. 页面调用现有 `addToCartAuth()`：登录用户写入 API 购物车，访客写入本地购物车。
6. 全局布局更新购物车角标。

本迭代不新增 Service、Repository、数据库表或迁移。商品详情读取逻辑足够简单，保留在控制器层符合当前项目模式。

## 7. 文件职责

- `routes/web.php`：声明公开商品详情路由和路由名称。
- `app/Http/Controllers/ProductController.php`：加载并授权公开商品详情。
- `app/Http/Controllers/Api/ProductController.php`：限制 API 详情仅返回已发布商品。
- `resources/views/shop/product-detail.blade.php`：渲染详情和数量加购交互。
- `resources/views/shop/catalog.blade.php`：为图片和标题增加详情链接。
- 三个 `resources/lang/*.json`：提供详情页三语文案。
- `tests/Feature/Web/ProductDetailTest.php`：验证 Web 详情行为与目录入口。
- `tests/Feature/Api/ProductApiDetailTest.php`：验证 API 发布状态边界。

## 8. 错误与边界处理

- 不存在商品：404。
- 未发布商品：404，不泄露其名称、价格或描述。
- 售罄商品：页面可查看，但不能加购。
- 数量小于 1 或大于库存：浏览器控件阻止常规输入；绕过前端时由购物车 API 验证。
- 图片为空：显示本地占位区域，不请求无效 URL。
- 可选字段为空：隐藏对应信息块。

## 9. 安全与数据正确性

- 商品详情不需要认证，但必须执行发布状态检查。
- 不在 Blade 中拼接未经转义的 HTML；商品文本使用 Blade 默认转义。
- 商品 ID、名称和价格传入 JavaScript 时分别使用整数转换、`Illuminate\Support\Js::from()` 和数值转换。
- 加购数量只作为客户端请求参数，库存判断继续以服务端为最终依据。

## 10. 测试策略

核心自动化测试必须覆盖：

1. 已发布商品详情返回 200，并显示名称、价格、描述和库存信息。
2. 草稿商品详情返回 404。
3. 归档商品详情返回 404。
4. 不存在商品返回 404。
5. 售罄商品显示售罄状态且加购按钮禁用。
6. 商品目录包含正确的详情链接。
7. API 已发布商品返回 200，并保持现有响应结构。
8. API 草稿与归档商品返回 404。

完成后运行详情页定向测试、相关购物车测试、完整 PHPUnit 测试和前端生产构建，并在本地浏览器手动验证桌面视图、窄屏视图及目录到详情的跳转。

## 11. 验收标准

- 用户点击目录中的商品图片或标题后进入对应详情页。
- 详情页可通过固定 URL 直接访问和刷新。
- 页面完整展示现有数据库中的核心商品信息。
- 用户可选择合法数量并完成加入购物车操作。
- 售罄商品无法加入购物车。
- 未发布商品不能通过 Web 或公开 API 查看。
- 新增文案在简体中文、繁体中文和英文环境均有对应值。
- 核心定向测试、完整测试套件和 Vite 构建全部通过。
- 功能可在本地 `http://127.0.0.1:8000` 完整演示。

## 12. 成功信号

- 目录中 100% 的已发布商品都有可用详情入口。
- 已发布商品详情请求成功率为 100%。
- 未发布商品公开访问阻断率为 100%。
- 新增用户界面文案三语覆盖率为 100%。
- 自动化测试回归失败数为 0。

## 13. 非目标与后续增强

以下能力不属于本迭代，不阻塞详情页交付：

- 商品评论与评分。
- 收藏或愿望清单。
- 相关推荐。
- 社交分享。
- 多图画廊和图片放大器。
- SEO 结构化数据。
- 商品内容多语言化。
- 独立库存预占或详情页实时库存推送。

## 14. 回滚策略

如本迭代引入回归，可移除详情路由和 Blade 页面，恢复目录图片与标题的非链接展示，并撤销 API 发布状态调整及新增测试、文案。数据库没有结构变化，无需数据库回滚。
