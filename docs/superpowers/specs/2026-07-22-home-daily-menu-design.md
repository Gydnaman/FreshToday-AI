# 首页每日菜单与登录态设计

## 目标

让已登录用户每天第一次进入首页时获得一份基于个人偏好和仓库可售商品生成的菜单。菜单按用户和日期持久化，当天重复访问保持不变，只有用户主动点击“重新生成”才替换当天菜单。首页可查看今天及过去 6 天的菜单。

本迭代同时修复两个直接影响演示的登录体验问题：已登录用户不再看到首页创建账户区域；普通登录成功后默认返回首页。

## 用户价值与成功信号

- 用户登录后可在首页直接看到当天个性化菜单，无需另找 dashboard。
- 同一用户在同一天刷新、退出重登或再次进入首页时看到同一菜单。
- 第二天第一次进入首页会生成新菜单；连续使用后可查看最近 7 天历史。
- 用户主动重新生成后，当天记录被替换；系统不自动改变当天已保存菜单。
- AI 不可用时仍能从数据库中的已上架、有库存商品生成可展示的本地降级菜单。
- 自动化测试、完整测试套件、生产构建和本地浏览器演示均通过。

## 当前状态

- `daily_menus` 已按 `user_id` 和 `date` 保存菜单，但数据库没有联合唯一约束。
- `AiMenuService::generateDailyMenuForUser()` 已具备“当天已有则返回”的基本能力。
- 每日 04:00 已有调度任务，为具备 `userPreferences` 的用户派发生成任务。
- `DashboardController` 只读取当天菜单；首页不读取菜单。
- `AiMenuService::regenerate()` 清除缓存后再次调用普通生成路径，仍可能命中当天数据库记录，因此并不保证真正重新生成。
- 本地 fallback 固定取前两个商品，同一偏好可能每天得到相同内容。
- 登录页未带 `return` 时默认跳转 `/catalog`。
- 首页创建账户区不区分登录状态。

## 必须实现的范围

### 1. 首页登录态

- 将 `/` 从闭包路由改为 `Web\HomeController::index`。
- 访客继续看到当前营销内容和创建账户区域。
- 已登录用户不渲染创建账户区域，并在首页渲染“今日菜单”区域。
- 菜单和历史数据必须只按当前登录用户的 `user_id` 查询，不允许读取其他用户数据。

### 2. 每日首次访问生成

- 已登录用户进入首页时，按应用配置时区计算当天日期，再查询“当前用户 + 今天”的菜单。
- 当天记录存在时直接返回，不调用 AI。
- 当天记录不存在且用户已有偏好时，同步调用 `AiMenuService` 生成、保存并显示。
- 当天记录不存在且用户没有偏好时，不调用 AI；显示完成偏好问卷的入口。
- 每日 04:00 调度任务继续保留。首页首次访问是补偿路径，使本地开发环境或调度延迟时仍能完成核心流程。

### 3. 每天不同但当天固定

- 可选商品必须同时满足 `status = published` 和 `stock > 0`。
- 商品按稳定 ID 排序后，使用“用户 ID + 当天日期”计算轮换起点，最多向模型发送 8 个商品名。
- 日期和轮换后的候选商品共同进入 prompt，使次日输入不同，同时控制输入 Token。
- DeepSeek 输出只能使用候选商品的精确名称；既有 `MenuOutputValidator` 继续校验该约束。
- 本地 fallback 使用相同日期种子和候选商品，生成早餐、午餐、晚餐的结构化菜单；不得发明数据库中不存在的商品。
- 当天普通生成使用已保存记录，因此随机温度不会导致刷新后改变内容。

### 4. 主动重新生成

- 首页按钮调用现有鉴权接口 `POST /api/menu/regenerate`。
- `AiMenuService` 增加显式强制生成路径；强制路径跳过当天缓存和数据库短路，但仍覆盖同一条当天记录。
- 成功后页面重新加载并显示新菜单。
- 保留现有每日最多 3 次重新生成限制。
- DeepSeek 失败但本地 fallback 成功时，以 fallback 覆盖当天菜单；限流、无可售商品或未处理异常等无法形成新菜单的情况保留旧菜单，UI 显示三语错误提示。

### 5. 最近 7 天菜单

- 首页固定提供从今天到过去第 6 天的 7 个日期入口，默认选中今天。
- 有记录的日期显示该日菜单；无记录的日期显示“当日暂无菜单”，不为过去日期补生成。
- `menu_json` 存在时使用 `MenuRenderer::renderHtmlFromJson()` 渲染，并保留商品链接。
- 仅有 `menu_content` 时显示经过 Blade 转义的纯文本。
- 历史记录按日期降序查询，限制在最近 7 天。

### 6. 登录回跳

- 登录页未提供 `?return=` 时，默认目标从 `/catalog` 改为 `/`。
- 显式 `?return=/checkout` 等目标继续优先，不能破坏结账登录回跳。
- 已登录用户再次访问 `/login` 时使用同一回跳规则。
- 显式回跳到结账等页面时不阻塞登录响应；菜单会在该用户下一次进入首页时补生成。

### 7. DeepSeek 配置与 Token 控制

- 密钥只从本地 `.env` 的 `DEEPSEEK_API_KEY` 读取，不写入 Git、规格、日志或测试快照。
- `AI_PROVIDER=deepseek` 明确选择 DeepSeek。
- 实际调用前通过 DeepSeek 模型列表接口确认账户可用的 V4 Flash model ID；若官方仅提供稳定别名，则使用 `deepseek-chat` 并记录验证结果。
- 每次仅发送最多 8 个候选商品名、必要的用户偏好和当天日期。
- 输出只包含问候、早餐、午餐、晚餐和一条提示；保持现有 JSON 结构，不增加推荐解释、价格或营销文案。
- 最大输出 Token 以能够稳定返回既有 JSON 契约的最小值为准，并通过一次真实调用验证。

## 数据设计

新增迁移，为 `daily_menus` 建立联合唯一索引：

```text
UNIQUE (user_id, date)
```

迁移前若存在非法重复数据，按每组 `user_id + date` 保留 ID 最大的最新记录，移除较旧重复记录，再创建唯一索引。回滚只移除唯一索引，不恢复无效重复记录。

业务写入继续覆盖同一条当天记录，不新增版本表。本迭代不保存重新生成历史。

## 组件与接口

### `Web\HomeController`

- 输入：当前 Web Session 用户。
- 访客输出：营销首页数据。
- 登录用户输出：当天菜单、最近 7 天菜单映射、每个菜单的安全 HTML 或纯文本展示数据。
- 依赖：`AiMenuService`、`DailyMenu`、`Product`、`MenuRenderer`。

### `AiMenuService`

- 普通生成：当天已有记录则直接返回。
- 强制生成：跳过缓存和数据库短路，生成成功后覆盖当天记录。
- 候选商品选择和本地 fallback 使用相同的日期轮换规则。
- API 异常、超时或无效输出进入本地 fallback。

### `Api\MenuController`

- 保留 `POST /api/menu/regenerate` 路由和鉴权。
- 成功返回当天新菜单元数据；失败保持既有错误契约。
- 不在响应或日志中暴露 prompt、API 密钥或用户敏感信息。

### 首页 Blade

- 访客区与登录菜单区互斥渲染。
- 菜单日期切换使用轻量原生 JavaScript，不引入新前端框架。
- 重新生成按钮复用 `gbFetch()`，显示 loading、成功后刷新、失败后恢复按钮并展示错误。
- 新增文案必须同时写入 `resources/lang/zh.json`、`zhhk.json`、`en.json`。

## 错误与边界处理

- 无用户偏好：显示问卷 CTA，不调用 AI。
- 没有可售商品：不请求 AI，显示“暂无可推荐商品”；不得构造虚假商品菜单。
- DeepSeek 非 2xx、超时、空响应或无效 JSON：生成并保存本地结构化 fallback。
- 同时打开多个首页请求：数据库唯一约束阻止同日重复记录；业务层在唯一冲突后重新读取当天记录。
- 重新生成超过每日限制：显示现有 `GUARD-AI-RATE` 错误，旧菜单保持不变。
- 历史某日缺失：只显示缺失状态，不回填、不调用 AI。
- 菜单内商品链接继续通过 `MenuRenderer` 转义非商品文本，避免 XSS。

## 测试设计

### 首页功能测试

- 访客看到创建账户区，不看到登录菜单区。
- 登录用户不看到创建账户区。
- 登录用户当天无菜单、有偏好时生成并显示一份菜单。
- 当天已有菜单时重复访问不再次调用 Provider。
- 无偏好用户看到问卷 CTA，Provider 不被调用。
- 最近 7 天仅显示当前用户数据，排除第 7 天之前和其他用户数据。
- `menu_json` 使用安全 HTML 渲染，纯文本菜单正确转义。

### 服务与 API 测试

- 同一天普通生成两次只调用 Provider 一次并返回同一记录。
- 日期推进一天后创建新记录，候选商品轮换。
- 主动重新生成再次调用 Provider并覆盖当天记录。
- 第 4 次重新生成继续返回限流错误。
- DeepSeek 失败时 fallback 只包含已上架、有库存商品，并随日期轮换。
- 并发或重复写入不能产生相同 `user_id + date` 的两条记录。

### 登录与浏览器测试

- 普通登录成功跳转 `/`。
- 带 `return=/checkout` 登录仍跳转 `/checkout`。
- 浏览器登录后创建账户区消失，今日菜单出现。
- 刷新页面菜单内容保持一致。
- 点击重新生成后内容或菜单记录更新时间发生变化。
- 切换最近 7 天日期可展示已有记录和缺失状态。
- 简体中文、繁体中文、英文均不显示裸翻译键。
- 移动端无横向溢出，日期入口、菜单卡片和重新生成按钮可操作。

## 非目标

- 不生成未来日期菜单。
- 不自动回填用户注册前或系统上线前的历史菜单。
- 不保存同一天多次重新生成的版本历史。
- 不新增收藏、评分、分享、营养计算、购物清单或一键下单。
- 不重写现有 dashboard；首页成为本迭代的主要菜单入口，dashboard 保持兼容。
- 不引入 Vue、React、Livewire、队列监控平台或新的 AI SDK。

## 完成标准

- 路由和首页 UI 可访问，登录后主流程可从登录页完整演示。
- 当日菜单首次生成后持久化，普通访问不重复调用 AI。
- 主动重新生成确实产生一次新生成并覆盖当天记录。
- 最近 7 天菜单严格按当前用户隔离。
- 未登录创建账户区和已登录菜单区互斥。
- DeepSeek 或本地 fallback 只使用仓库内已上架、有库存商品。
- 三语文案、核心自动化测试、完整 PHPUnit、Vite 构建和浏览器验收通过。

## 风险与回滚

- 首次访问可能等待 DeepSeek 请求；通过 04:00 预生成和较短超时降低影响，失败立即使用本地 fallback。
- V4 Flash 的公开 model ID 可能随账户或平台别名变化；在写入配置前先调用模型列表验证。
- 回滚应用代码后可继续保留联合唯一索引，因为它符合原有“一用户一天一份菜单”的业务语义；如必须完全回滚，执行迁移 `down()` 移除索引。
